## Что НЕ сделано и что нужно сделать

### 1) Production-конфигурация маршрутизации менеджеров

Сделано:

- Расширен seed в `admin-panel`: теперь `manager2LDM` назначен на ВСЕ 8 категорий (не только Steel и Aluminium).
- Добавлены пример-назначения на конкретные продукты для демонстрации product-level routing.
- Добавлены категории и менеджер-назначения для Company2, чтобы каждый tenant имел полную конфигурацию.

Процесс конфигурирования в production:

1. При первом запуске seed создаёт baseline-назначения для всех менеджеров и категорий.
2. Admin может затем через UI (`/admin/manager-assignments`) уточнять и переопределять назначения:
   - Выбрать менеджера.
   - Отметить категории, за которые он отвечает (или включить `handlesAllCategories` для универсального менеджера).
   - Опционально назначить товары, требующие специальной маршрутизации.
   - Установить `isPrimary` для главной категории/товара, на которых должны фокусироваться заявки.
3. Система гарантирует fallback: если заявка не покрыта конкретным товаром/категорией, она идёт к универсальному менеджеру.

Актуальный код:

- [admin-panel/app/admin/manager-assignments/page.tsx](admin-panel/app/admin/manager-assignments/page.tsx) — админ-UI.
- [admin-panel/app/api/manager-assignments/route.ts](admin-panel/app/api/manager-assignments/route.ts) — API для сохранения.
- [admin-panel/prisma/seed.ts](admin-panel/prisma/seed.ts) — baseline seed (линия ~470-550).

### 3) Безопасная отвязка ингредиентов и перевод в отдельные товары

Текущее состояние:

- Runtime flow переведен на товарную модель: cart, order, search и product-form больше не зависят от `ingredients`.
- Seed обновлен: бывшие ингредиенты сидируются как обычные товары категории `Крепеж`.
- Legacy UI/runtime хвосты (`item.ingredients`, `IngredientItem`, старый backfill script) удалены.
- В рабочем коде ingredient-упоминания остались только в историческом baseline migration SQL.

Что еще осталось довести:

- Зафиксировано процессное правило: `0000_baseline` в `ldm-steel` считается архивным артефактом и не является источником актуальной схемы.
- Источник истины для Prisma-схемы и миграций находится в `admin-panel`, где уже есть миграция удаления ingredient-таблиц.
- Отдельная перегенерация historical baseline в `ldm-steel` не требуется, чтобы не переписывать историю миграций.

Что нужно сделать (по этапам):

#### Этап A: Bridge-слой (безопасный, additive)

1. Перевести доп. позиции на модель отдельных товаров (категория может быть любой: крепеж, профиль, фурнитура, ручки, петли и т.д.).
2. Добавить универсальную связь "с этим товаром берут" между товарами (любой товар -> любой товар).
3. На этом этапе не удалять legacy-связи ingredients, пока не завершен перенос.
4. Цена остается в товаре (варианте товара), а показ цены управляется через `priceMode` в admin-panel.

##### Этап A: задачи по PR (рекомендуемый порядок)

PR-A1 (Schema: related products + price policy):

1. Добавить в Prisma универсальную связь между товарами для блока "с этим товаром берут" (например, `ProductRelatedProduct`), без ограничения на конкретные категории.
